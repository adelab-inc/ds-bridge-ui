**자연스럽게 본문 안에 흡수해서** 다시 쓴 버전이야.

# **2막. 실행이 ‘편집’으로 넘어간다 (Day 4–6)**

---

## **이 단계에서 바뀌는 건 단 하나**

> “보기만 하던 디자인 시스템을
> 

> 직접 ‘만져볼 수 있게’ 된다.”
> 

1막에서 사용자는 이미 확신했다.

- “여긴 진짜다”
- “실제 디자인 시스템이 실행된다”

그래서 자연스럽게 다음 질문이 나온다.

> “그럼 이걸
> 

> 내가 원하는 상태로도 바꿔볼 수 있어?”
> 

2막은 이 질문에 답하는 단계다.

---

## **먼저, 오해부터 분명히 하자**

이 질문에 답하기 전에,

2막에서 **우리가 하지 않는 것**부터 분명히 해야 한다.

- ❌ 컴포넌트 코드를 수정하지 않는다
- ❌ 디자인 시스템(JSON)을 고치지 않는다
- ❌ iframe 안의 화면을 직접 클릭해서 편집하지 않는다

즉,

> 2막은 ‘정의’를 바꾸는 단계가 아니다.
> 

디자인 시스템은 그대로 두고,

**그 위에서 실행되고 있는 상태만 다룬다.**

---

## **그럼 2막에서 뭘 바꾸는 걸까?**

> “지금 화면에 올라가 있는
> 

> 구성 요소 하나의 실행 상태”
> 

이걸 바꾼다.

여기서 말하는 실행 상태란,

- 이 버튼이 어떤 variant인지
- 어떤 size인지
- loading 상태인지 아닌지

같은 **지금 이 순간의 값**이다.

---

## **실제 화면 구조 (1막에서 확장된다)**

1막의 화면이 이렇게 생겼다면,

```
┌───────────────────────────┬───────────────────────────┐
│  컴포넌트 목록            │  실행 화면                │
│                           │  (Storybook iframe)       │
│  Button                   │                           │
│  TextField                │   실제 실행 중인 UI       │
│  Checkbox                 │                           │
│                           │                           │
└───────────────────────────┴───────────────────────────┘
```

2막에서는 여기에 **두 가지가 추가된다.**

```
┌───────────────────────────┬───────────────────────────┬───────────────┐
│  구성 트리                │  실행 화면                │  Props 패널   │
│                           │  (iframe)                 │               │
│  ▸ Login Page             │                           │               │
│    ├─ TextField (Email)   │   [ Email Input ]         │               │
│    ├─ TextField (PW)      │   [ Password Input ]      │               │
│    └─ Button (Login)      │   [  Login Button  ]      │               │
│                           │                           │               │
└───────────────────────────┴───────────────────────────┴───────────────┘
```

- 왼쪽: **지금 화면이 어떻게 구성돼 있는지 보여주는 구조**
- 가운데: **실제 실행 결과**
- 오른쪽: **선택한 구성 요소의 상태를 바꾸는 패널**

---

## **여기서 가장 중요한 원칙**

> 편집은 항상
> 

> ‘구성 트리 + Props 패널’에서만 한다.
> 

> 
> 

> 실행 화면은 결과만 보여준다.
> 

iframe 안의 UI는

**절대 직접 만지지 않는다.**

이걸 지키지 않으면:

- “내가 바꾼 건지, 시스템이 바꾼 건지” 헷갈리고
- 나중에 AI가 개입할 여지가 사라진다.

---

## **“버튼이 여러 개 있으면?”에 대한 정확한 답**

여기서 가장 많이 나오는 질문이 이거다.

> “페이지에 버튼이 여러 개 있으면
> 

> 그럼 뭘 클릭해서 뭘 바꾸는 거야?”
> 

예를 들어 버튼이 두 개 있다.

- 위: Submit 버튼
- 아래: Cancel 버튼

시스템은 이걸 이렇게 인식한다.

```
Page
 ├─ Button#1 (Submit)
 └─ Button#2 (Cancel)
```

- 둘 다 같은 Button 컴포넌트
- 하지만 **서로 다른 인스턴스**

---

## **사용자가 실제로 하는 행동**

### **1️⃣ 구성 트리에서 하나를 선택한다**

```
▸ Button#1 (Submit)
```

이걸 클릭한다.

> 중요:
> 

> 실행 화면 안의 버튼을 클릭하는 게 아니다.
> 

> 구성 트리에서 ‘이 버튼’을 선택한다.
> 

---

### **2️⃣ Props 패널이 열린다**

```
[ Button (Submit) - Props ]

variant:  [ primary ▼ ]
size:     [ md ▼ ]
loading:  [ off ]
```

이 패널은 자동으로 만들어진다.

- 디자인 시스템(JSON)에 정의된 props를 읽고
- 가능한 값만
- 가장 단순하고 직관적인 UI로 보여준다

---

## **여기서 “편집”이란 정확히 무엇인가**

사용자가 값을 바꾼다.

- variant: primary → secondary
- size: md → lg

이 순간 벌어지는 일은 이것뿐이다.

```
// Button#1 인스턴스의 실행 상태
args = {
  variant: "secondary",
  size: "lg"
}
```

그리고 이 args가

**Storybook iframe으로 전달된다.**

---

## **그 결과 화면은 어떻게 바뀔까?**

- Submit 버튼만
    - 색이 바뀌고
    - 크기가 바뀐다
- Cancel 버튼은 **아무 변화도 없다**

왜냐하면 우리는:

> Button이라는 정의를 바꾼 게 아니라
> 

> Button#1이라는 인스턴스 하나의 실행 상태만 바꿨기 때문
> 

---

## **그럼 채팅 UI는 여기서 뭘 하느냐**

2막에서 채팅은 **조용한 보조자**다.

채팅은 화면을 바꾸지 않는다.

채팅은 대신 이렇게 말한다.

```
Assistant:
You’re editing Button (Login).

Current state:
- variant: primary
- size: md

You can:
• Change variant
• Adjust size
• Toggle loading state
```

즉, 채팅은:

- ❌ 대신 바꾸지 않고
- ❌ 결과를 강요하지 않고
- **지금 사용자가 무엇을 만지고 있는지를 말로 정리해준다**

> 2막에서 채팅은
> 

> ‘손’이 아니라 ‘해설자’다.
> 

---

## **왜 이 방식이 중요한가**

이 방식 덕분에 다음이 가능해진다.

- 버튼이 여러 개 있어도
    
    **어느 버튼을 바꾸는지 명확하다**
    
- 실행 결과를 링크로 공유해도
    
    **같은 상태로 정확히 재현된다**
    
- AI가 나중에 이렇게 말할 수 있다

> “로그인 버튼(Button#1)을
> 

> secondary로 바꿔줘”
> 

즉,

> ‘편집’의 단위가 처음부터 끝까지 흔들리지 않는다.
> 

---

## **2막에서 사용자가 느끼는 변화**

이 시점에서 사용자는 이렇게 말한다.

> “아, 이 디자인 시스템이
> 

> 어떤 범위 안에서 움직이는지 이제 알겠다.”
> 

> 
> 

> “문서 읽을 필요 없네.”
> 

> 
> 

> “여기서 다 만져보고 결정하면 되겠다.”
> 

이 순간, 이 플랫폼은:

- 문서 ❌
- 뷰어 ❌
- **워크플로우 툴 ⭕**

이 된다.

---

## **성공 기준 (Definition of Done)**

2막은 아래가 되면 끝이다.

- 구성 트리에서 컴포넌트를 선택할 수 있다
- props를 바꾸면 화면이 즉시 반영된다
- 여러 인스턴스 중 선택한 것만 바뀐다
- JSON과 컴포넌트 코드는 바뀌지 않는다

---

## **2막의 역할**

2막은 생산성을 높이는 단계가 아니다.

**2막은 이해를 완성하는 단계다.**

> “이 디자인 시스템은
> 

> 이렇게 움직이고,
> 

> 이 정도까지 조합할 수 있구나.”
> 

이 감각이 생겨야

다음 단계가 의미를 갖는다.
