# 2막. 실행이 '편집'으로 넘어간다

---

## 이 단계에서 바뀌는 건 단 하나

> **"보기만 하던 디자인 시스템을 직접 '만져볼 수 있게' 된다."**

1막에서 사용자는 이미 확신했다:
- "여긴 진짜다"
- "실제 디자인 시스템이 실행된다"

2막은 그 다음 질문에 답한다:

> "그럼 이걸 내가 원하는 상태로도 바꿔볼 수 있어?"

---

## 먼저, 오해부터 정리하자

2막에서 **하지 않는 것**:
- ❌ 컴포넌트 코드를 수정하지 않는다
- ❌ 디자인 시스템(JSON)을 고치지 않는다
- ❌ iframe 안의 화면을 직접 클릭해서 편집하지 않는다

**즉, '정의'를 바꾸는 단계가 아니다.**

---

## 그럼 뭘 바꾸는 걸까?

> **"지금 화면에 올라가 있는 구성 요소 하나의 상태"**

이걸 바꾼다.

---

## 실제 화면 구조

```
┌───────────────────────────┬───────────────────────────┐
│  왼쪽: 구성 트리          │  오른쪽: 실행 화면        │
│                           │  (Storybook iframe)       │
│  ▸ Login Page             │                           │
│    ├─ TextField (Email)   │   [ Email Input ]         │
│    ├─ TextField (PW)      │   [ Password Input ]      │
│    └─ Button (Login)      │   [  Login Button  ]      │
│                           │                           │
└───────────────────────────┴───────────────────────────┘
```

- 오른쪽은 결과 화면
- 왼쪽은 지금 화면이 어떻게 구성되어 있는지 보여주는 구조

**중요한 점:**
편집은 항상 왼쪽(또는 옆 패널)에서만 한다.
결과 화면은 절대 직접 건드리지 않는다.

---

## "버튼이 여러 개 있으면?"

예를 들어 화면에 버튼이 두 개 있다:
- 위: Submit 버튼
- 아래: Cancel 버튼

시스템은 이걸 이렇게 인식한다:

```
Page
 ├─ Button#1 (Submit)
 └─ Button#2 (Cancel)
```

두 버튼은 **같은 컴포넌트(Button)**지만 **서로 다른 인스턴스**다.

---

## 사용자가 실제로 하는 행동

### 1️⃣ 구성 트리에서 하나를 선택한다

```
▸ Button#1 (Submit)
```

이걸 클릭한다.

### 2️⃣ Props 패널이 열린다

```
[ Button (Submit) - Props ]

variant:  [ primary ▼ ]
size:     [ md ▼ ]
loading:  [ off ]
```

이 패널은 자동으로 만들어진다:
- 디자인 시스템(JSON)에 정의된 props를 읽어서
- 가장 단순하고 직관적인 UI로 보여준다

---

## 여기서 "편집"이란 정확히 무엇인가

사용자가 값을 바꾼다:
- variant: `primary` → `secondary`
- size: `md` → `lg`

이 순간 벌어지는 일:

```javascript
// Button#1 인스턴스의 실행 상태
args = {
  variant: "secondary",
  size: "lg"
}
```

그리고 이 args가 Storybook iframe으로 전달된다.

---

## 그 결과 화면은?

- Submit 버튼만 색이 바뀌고 크기가 바뀐다
- Cancel 버튼은 아무 변화 없다

**왜냐하면:**

> 우리는 Button이라는 정의를 바꾼 게 아니라
> Button#1이라는 인스턴스 하나의 실행 상태만 바꿨기 때문이다.

---

## 왜 이 방식이 중요한가

이 방식 덕분에 다음이 가능해진다:
- 버튼이 여러 개 있어도 "어느 버튼을 바꾸는지"가 명확하다
- 실행 결과를 링크로 공유해도 재현된다
- AI가 나중에 이렇게 말할 수 있다: "로그인 버튼(Button#1)을 primary로 바꿔줘"

**즉, '편집'의 단위가 명확해진다.**

---

## 2막에서 사용자가 느끼는 변화

이 시점에서 사용자는 이렇게 말한다:

> "아, 이 디자인 시스템이 어떤 범위 안에서 동작하는지 이제 알겠다."
> "문서 읽을 필요 없네."
> "여기서 다 만져보고 결정하면 되겠다."

이 순간, 이 플랫폼은:
- 문서 ❌
- 뷰어 ❌
- **워크플로우 툴 ⭕**

이 된다.

---

## 성공 기준 (Definition of Done)

- [ ] 구성 트리에서 컴포넌트를 선택할 수 있다
- [ ] props를 바꾸면 화면이 즉시 반영된다
- [ ] 여러 인스턴스 중 선택한 것만 바뀐다
- [ ] JSON과 코드가 바뀌지 않는다

---

## 2막의 역할

2막은 생산성을 높이는 단계가 아니다.
**2막은 이해를 완성하는 단계다.**

> "이 디자인 시스템은 이렇게 움직이고, 이 정도까지 조합할 수 있구나."

이 감각이 생겨야 다음 단계가 의미를 갖는다.
