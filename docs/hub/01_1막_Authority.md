# 1막. Authority를 빌려서, 신뢰를 만든다 

## **이 단계의 목표**

> “이 서비스는 진짜 디자인 시스템을 다룬다.”
> 

> 이 인상을
> 
> 
> **10초 안에**
> 

아무리 좋은 아이디어라도,

처음 만나는 사람의 머릿속에는 늘 같은 의문이 떠오른다.

- “이거 데모 아닌가?”
- “실험용 툴 아니야?”
- “우리 회사에 쓰기엔 아직 불안한데…”

**그래서 우리는 설명부터 하지 않는다.**

대신, 이미 모두가 아는 디자인 시스템을 **바로 실행**시킨다.

---

## **사용자가 처음 보게 되는 화면**

첫 화면에는 복잡한 설정도, 긴 설명도 없다.

딱 이 세 가지 선택지만 보인다.

```
[ Run MUI ]
[ Run Ant Design ]
[ Run Carbon ]
```

이 버튼들은 예제가 아니다.

**실제 운영 중인 디자인 시스템**이다.

---

## **버튼을 누르면 무슨 일이 벌어질까?**

예를 들어 Run MUI를 누른다.

화면은 이렇게 바뀐다.

```
┌───────────────────────────┬───────────────────────────┐
│  컴포넌트 목록            │  실행 화면                │
│                           │  (Storybook iframe)       │
│  Button                   │                           │
│  TextField                │   실제 MUI 버튼           │
│  Checkbox                 │   실제 MUI 입력창         │
│  Dialog                   │                           │
│                           │                           │
└───────────────────────────┴───────────────────────────┘
```

- 왼쪽에는 컴포넌트 이름 목록
- 오른쪽에는 **실제로 동작하는 UI**

이 화면은:

- 이미지가 아니다
- 설명도 아니다
- **진짜 실행 중인 UI다**

---

## **이때, 채팅이 처음 등장한다**

이 시점에서 채팅은 **무언가를 만들어주지 않는다.**

대신 이렇게 말을 건다.

```
Assistant:
MUI Design System is loaded.

You can:
• Browse components
• Run a component
• See available states (stories)

What would you like to explore?
```

중요한 점은 이거다.

> 채팅은 추측하지 않는다.
> 

> 채팅은 JSON에 있는 정보만 말한다.
> 

---

## **채팅은 ‘안내자’ 역할만 한다**

사용자가 이렇게 물을 수 있다.

```
User:
버튼에는 뭐가 있어?
```

채팅은 이렇게 답한다.

```
Assistant:
MUI Button component has these variants:

• Primary
• Secondary
• Outline
• Destructive

You can click any variant to run it.
```

그리고 화면에는

**실제 실행 가능한 선택지**가 같이 나타난다.

채팅의 모든 응답은

**항상 실행으로 이어진다.**

---

## **이 단계에서 “편집”은 없다**

중요한 점을 분명히 하자.

이 단계에서는:

- 아무 것도 바꾸지 않는다
- props를 조작하지 않는다
- 설정을 만질 수 없다
- iframe 안을 직접 건드리지 않는다

**이 단계의 역할은 단 하나, ‘증명’이다.**

> “이 플랫폼은
> 

> 스토리북을 흉내 내는 게 아니라
> 

> 실제로 스토리북을 실행한다.
> 

---

## **왜 굳이 남의 디자인 시스템을 쓰는가?**

우리는 처음부터

우리만의 디자인 시스템을 보여주지 않는다.

대신 이미 업계에서 검증된 시스템을 쓴다.

- MUI
- Ant Design
- Carbon

이 이름들은

긴 설명이 필요 없다.

사용자는 화면을 보는 순간 이렇게 느낀다.

- “아, 이건 연습용 예제가 아니구나.”
- “진짜 실무에서 쓰는 것들이네.”
- “그럼 우리 것도 될 수 있겠네.”

**이게 바로 Authority를 빌리는 이유다.**

---

## **페르소나별로 이 단계가 주는 신호**

### **개발자에게**

- “스토리북 URL 하나로 여기서 실행되네”
- “컴포넌트 구조가 한눈에 보이네”
- “코드를 안 봐도 범위가 감이 온다”

→ **신뢰 가능한 도구인지 판단하는 단계**

---

### **디자이너에게**

- “실제 결과를 바로 볼 수 있네”
- “문서 없이도 감이 온다”
- “코드 몰라도 이해할 수 있겠다”

→ **내가 써도 되는 도구인지 판단하는 단계**

---

### **일반 사용자에게**

> “아, 이 서비스는
> 

> 디자인 시스템을 모아놓고
> 

> 바로 돌려보게 해주는 곳이구나.”
> 

복잡한 설명 없이도

무슨 서비스인지 바로 이해된다.

---

## **성공 기준 (Definition of Done)**

1막이 성공했다는 건 기능이 많다는 뜻이 아니다.

아래 세 가지가 되면 끝이다.

- MUI 같은 유명 디자인 시스템이 즉시 실행된다
- 컴포넌트를 클릭하면 실제 화면이 보인다
- 사용자가 설명 없이도 “뭘 하는 곳인지” 이해한다

---

## **1막의 역할**

이 단계에서 우리는:

- 문제를 풀지 않는다
- 생산성을 약속하지 않는다
- AI를 자랑하지 않는다

**단 하나만 한다.**

> “이건 장난이 아니다.”
> 

이 신뢰가 생겼을 때,

비로소 다음 단계로 넘어갈 수 있다.
