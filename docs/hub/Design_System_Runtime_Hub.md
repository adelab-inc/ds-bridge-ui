우리가 만드는 첫 경험

우리가 만드는 건 디자인 시스템 문서 사이트가 아니다.
디자인 시스템을 직접 실행해보는 화면이다.
	•	Storybook은 컴포넌트를 실행할 수 있는 곳
	•	하지만 “조합해보고, 비교하고, 결정하는” 흐름은 없다

우리는 이걸 만든다:

디자인 시스템을 읽지 않고, 바로 돌려보는 환경

구조는 이렇게 나뉜다.
	•	JSON:
컴포넌트가 무엇인지, 어떤 props가 있는지에 대한 정의
	•	Storybook:
그 정의를 실제로 그려주는 실행기

👉 우리는 JSON과 Storybook을 연결해서
**“실행 가능한 디자인 시스템”**을 만든다.



알겠다.
아래는 1막만 따로 떼어서,
개발자·디자이너·일반인 누구라도 바로 이해할 수 있게,
전문적인데 어렵지 않게,
실제 화면에서 무슨 일이 벌어지는지 중심으로 쓴 설명이다.

1막. Authority를 빌려서, 신뢰를 만든다 (Day 1–3)

이 단계의 목표는 딱 하나다

“이 서비스는 진짜 디자인 시스템을 다룬다.”
이 인상을 10초 안에 주는 것.

아무리 좋은 아이디어라도,
처음 만나는 사람은 이렇게 생각한다.
	•	“이거 데모 아닌가?”
	•	“실험용 툴 아니야?”
	•	“우리 회사에 쓰기엔 불안한데…”

그래서 우리는 설명부터 하지 않는다.
대신 이미 모두가 아는 디자인 시스템을 바로 실행시킨다.

⸻

사용자가 처음 보게 되는 화면

첫 화면에는 복잡한 설정도, 긴 설명도 없다.
딱 이 세 가지가 보인다.

[ Run MUI ]
[ Run Ant Design ]
[ Run Carbon ]

이 버튼들은 “예제”가 아니다.
실제 운영 중인 디자인 시스템이다.

⸻

버튼을 누르면 무슨 일이 벌어질까?

예를 들어 Run MUI를 누른다.

화면이 이렇게 바뀐다.

┌───────────────────────────┬───────────────────────────┐
│  컴포넌트 목록            │  실행 화면                │
│                           │  (Storybook)              │
│  Button                   │                           │
│  TextField                │   실제 MUI 버튼           │
│  Checkbox                 │   실제 MUI 입력창         │
│  Dialog                   │                           │
│                           │                           │
└───────────────────────────┴───────────────────────────┘

왼쪽에는 컴포넌트 이름들,
오른쪽에는 실제로 동작하는 화면이 보인다.

이 화면은 이미지가 아니다.
설명이 아니다.
진짜 실행 중인 UI다.

⸻

이 단계에서 “편집”은 없다

중요한 점은 이거다.
	•	아직 아무것도 바꾸지 않는다
	•	버튼을 눌러도 모양이 바뀌지 않는다
	•	설정을 건드릴 수 없다

이 단계의 역할은 증명이다.

“이 플랫폼은
스토리북을 흉내 내는 게 아니라
실제로 스토리북을 실행한다.”

⸻

왜 굳이 남의 디자인 시스템을 쓰는가?

우리는 처음부터 우리만의 디자인 시스템을 보여주지 않는다.
대신 이미 업계에서 검증된 시스템을 쓴다.
	•	MUI
	•	Ant Design
	•	Carbon

이 이름들은 설명이 필요 없다.

사용자는 화면을 보는 순간 이렇게 느낀다.

“아, 이건 연습용 예제가 아니구나.”
“진짜 실무에서 쓰는 것들이네.”
“그럼 우리 것도 될 수 있겠네.”

이게 바로 Authority를 빌리는 이유다.

	•	“코드를 안 봐도 어떤 컴포넌트가 있는지 알겠네”

	•	“이게 실제로 어떻게 보이는지 바로 확인되네”
	•	“문서 읽지 않아도 느낌이 온다”
	•	“코드 몰라도 구조를 이해할 수 있겠다”

👉 ‘내가 써도 되는 도구인지’ 판단하는 단계

⸻

일반 사용자에게는 어떻게 보일까?

개발도 디자인도 잘 모르는 사람에게는 이렇게 보인다.

“아, 이 서비스는
디자인 시스템을 모아놓고
바로 돌려보게 해주는 곳이구나.”

복잡한 설명 없이도
무슨 서비스인지 바로 이해된다.

⸻

이 단계의 성공 기준 (Definition of Done)

1막이 성공했다는 건, 기능이 많은 게 아니다.
아래 세 가지가 되면 끝이다.
	•	MUI 같은 유명 디자인 시스템이 즉시 실행된다
	•	컴포넌트를 클릭하면 실제 화면이 보인다
	•	사용자가 설명 없이도 “뭘 하는 곳인지” 이해한다

이게 되면,
사람들은 더 이상 이 서비스를 의심하지 않는다.

1막의 역할은 여기까지다

이 단계에서 우리는:
	•	문제를 풀지 않는다
	•	생산성을 약속하지 않는다
	•	AI를 자랑하지 않는다

단 하나만 한다.

“이건 장난이 아니다.”

개발자·디자이너·일반인 모두 이해할 수 있게,
전문적이지만 어렵지 않게 쓴 설명이다.

2막. 실행이 ‘편집’으로 넘어간다 (Day 4–6)

이 단계에서 바뀌는 건 단 하나다

“보기만 하던 디자인 시스템을
직접 ‘만져볼 수 있게’ 된다.”

1막에서 사용자는 이미 확신했다.
	•	“여긴 진짜다”
	•	“실제 디자인 시스템이 실행된다”

2막은 그 다음 질문에 답한다.

“그럼 이걸 내가 원하는 상태로도 바꿔볼 수 있어?”

⸻

먼저, 오해부터 정리하자

2막에서 하지 않는 것부터 분명히 한다.
	•	❌ 컴포넌트 코드를 수정하지 않는다
	•	❌ 디자인 시스템(JSON)을 고치지 않는다
	•	❌ iframe 안의 화면을 직접 클릭해서 편집하지 않는다

즉,
‘정의’를 바꾸는 단계가 아니다.

⸻

그럼 뭘 바꾸는 걸까?

“지금 화면에 올라가 있는 구성 요소 하나의 상태”

이걸 바꾼다.

⸻

실제 화면 구조부터 보자

┌───────────────────────────┬───────────────────────────┐
│  왼쪽: 구성 트리          │  오른쪽: 실행 화면        │
│                           │  (Storybook iframe)       │
│  ▸ Login Page             │                           │
│    ├─ TextField (Email)   │   [ Email Input ]         │
│    ├─ TextField (PW)      │   [ Password Input ]      │
│    └─ Button (Login)      │   [  Login Button  ]      │
│                           │                           │
└───────────────────────────┴───────────────────────────┘

	•	오른쪽은 결과 화면
	•	왼쪽은 지금 화면이 어떻게 구성되어 있는지 보여주는 구조

중요한 점:

편집은 항상 왼쪽(또는 옆 패널)에서만 한다.
결과 화면은 절대 직접 건드리지 않는다.

⸻

“버튼이 여러 개 있으면?”에 대한 정확한 답

예를 들어 화면에 버튼이 두 개 있다.
	•	위: Submit 버튼
	•	아래: Cancel 버튼

시스템은 이걸 이렇게 인식한다.

Page
 ├─ Button#1 (Submit)
 └─ Button#2 (Cancel)

두 버튼은 **같은 컴포넌트(Button)**지만
서로 다른 인스턴스다.

⸻

사용자가 실제로 하는 행동

1️⃣ 구성 트리에서 하나를 선택한다

▸ Button#1 (Submit)

이걸 클릭한다.

2️⃣ 그러면 Props 패널이 열린다

[ Button (Submit) - Props ]

variant:  [ primary ▼ ]
size:     [ md ▼ ]
loading:  [ off ]

이 패널은 자동으로 만들어진다.
	•	디자인 시스템(JSON)에 정의된 props를 읽어서
	•	가장 단순하고 직관적인 UI로 보여준다

⸻

여기서 “편집”이란 정확히 무엇인가

사용자가 값을 바꾼다.
	•	variant: primary → secondary
	•	size: md → lg

이 순간 벌어지는 일은 이것뿐이다.

// Button#1 인스턴스의 실행 상태
args = {
  variant: "secondary",
  size: "lg"
}

그리고 이 args가
Storybook iframe으로 전달된다.

⸻

그 결과 화면은 어떻게 되나
	•	Submit 버튼만
	•	색이 바뀌고
	•	크기가 바뀐다
	•	Cancel 버튼은 아무 변화 없다

왜냐하면:

우리는 Button이라는 정의를 바꾼 게 아니라
Button#1이라는 인스턴스 하나의 실행 상태만 바꿨기 때문이다.

⸻

왜 이 방식이 중요한가

이 방식 덕분에 다음이 가능해진다.
	•	버튼이 여러 개 있어도
“어느 버튼을 바꾸는지”가 명확하다
	•	실행 결과를 링크로 공유해도 재현된다
	•	AI가 나중에 이렇게 말할 수 있다
“로그인 버튼(Button#1)을 primary로 바꿔줘”

즉,

‘편집’의 단위가 명확해진다.

⸻

2막에서 사용자가 느끼는 변화

이 시점에서 사용자는 이렇게 말한다.

“아, 이 디자인 시스템이
어떤 범위 안에서 동작하는지 이제 알겠다.”

“문서 읽을 필요 없네.”

“여기서 다 만져보고 결정하면 되겠다.”

이 순간, 이 플랫폼은:
	•	문서 ❌
	•	뷰어 ❌
	•	워크플로우 툴 ⭕

이 된다.

⸻

2막의 성공 기준 (Definition of Done)

이 단계는 아래가 되면 끝이다.
	•	구성 트리에서 컴포넌트를 선택할 수 있다
	•	props를 바꾸면 화면이 즉시 반영된다
	•	여러 인스턴스 중 선택한 것만 바뀐다
	•	JSON과 코드가 바뀌지 않는다

⸻

2막의 역할은 여기까지다

2막은 생산성을 높이는 단계가 아니다.
2막은 이해를 완성하는 단계다.

“이 디자인 시스템은
이렇게 움직이고,
이 정도까지 조합할 수 있구나.”

이 감각이 생겨야
다음 단계가 의미를 갖는다.

⸻

다음 질문은 항상 이거다.

“그럼 이제 이걸
AI가 대신 해줄 수 있는 거야?”

그 질문에 답하는 게 3막이다.




좋아.
아래는 3막만 단독으로,
앞의 1·2막을 이미 이해한 상태에서,
실제 화면 기준 / 헷갈리지 않게 / 톤 유지해서 쓴 설명이다.

⸻

3막. AI가 ‘생성’이 아니라 ‘조합’을 한다 (Day 7–10)

이 단계의 전제부터 분명히 하자

3막은 갑자기 AI가 똑똑해지는 단계가 아니다.
3막은 사람이 이미 이해한 걸 AI가 대신 해주는 단계다.

2막까지 사용자는 이미 이것을 손으로 느꼈다.
	•	이 디자인 시스템에 어떤 컴포넌트가 있는지
	•	어떤 props가 가능한지
	•	어떤 조합이 자연스러운지
	•	어떤 조합은 안 되는지

그래서 이제 질문이 바뀐다.

“이걸 내가 매번 직접 하지 말고,
AI가 대신 해줄 수는 없을까?”

⸻

먼저, AI가 하지 않는 것

3막에서 AI는 다음을 하지 않는다.
	•	❌ 새로운 UI를 상상해서 그리지 않는다
	•	❌ 디자인 시스템에 없는 컴포넌트를 만들지 않는다
	•	❌ props를 마음대로 invent하지 않는다

즉,

AI는 ‘창작자’가 아니다.
AI는 ‘조합자’다.

⸻

그럼 AI는 정확히 무슨 일을 할까?

AI가 하는 일은 딱 이거다.

“이 디자인 시스템 안에 이미 있는 것들 중에서
지금 목적에 맞는 인스턴스들을 골라서
페이지 구조를 만들어준다.”

⸻

실제 화면에서 벌어지는 흐름

1️⃣ 사용자가 한 줄을 입력한다
화면 상단에 입력창이 있다.

What do you want to build?

> 로그인 페이지 만들어줘


⸻

2️⃣ AI는 JSON을 본다 (전체가 아니라 일부만)
AI는 다음을 본다.
	•	이 DS에 어떤 컴포넌트들이 있는지
	•	각 컴포넌트에 어떤 props가 있는지
	•	어떤 story들이 대표적인지

중요한 점:
AI는 전체 JSON을 다 읽지 않는다.
“로그인”과 관련된 컴포넌트만 슬라이스해서 본다.

⸻

3️⃣ AI의 결과는 ‘화면’이 아니라 ‘조합’이다
AI가 내놓는 결과는 JSX가 아니다.
이미지도 아니다.

이런 조합 데이터다.

Login Page
 ├─ TextField (Email)
 ├─ TextField (Password)
 └─ Button (Primary)

각 항목은 이렇게 명확하다.
	•	어떤 컴포넌트인지
	•	어떤 story를 쓰는지
	•	어떤 props 상태인지

⸻

4️⃣ 이 조합이 바로 실행된다
이 조합은 곧바로 화면에 올라간다.

┌───────────────────────────┐
│ [ Email Input ]           │
│ [ Password Input ]        │
│ [  Login Button  ]        │
└───────────────────────────┘

여기서 중요한 점:
	•	이 화면은 AI가 그린 그림이 아니다
	•	스토리북에 이미 존재하던 컴포넌트 조합이다

⸻

그래서 3막의 감각은 이렇다

사용자는 이렇게 느낀다.

“AI가 새 걸 만들어준 게 아니라”
“우리 시스템에서 맞는 걸 골라줬네”

“그러면 이 결과는 믿을 수 있겠다”

이게 ‘생성’과 ‘조합’의 차이다.

⸻

3막에서 편집은 어떻게 이어지나?

여기서 2막과 자연스럽게 이어진다.

AI가 만든 화면은 끝이 아니다.
시작점이다.

사용자는 다시 할 수 있다.
	•	구성 트리에서 Button 선택
	•	variant 바꾸기
	•	size 바꾸기
	•	loading 켜보기

즉,

AI → 기본 조합
사람 → 미세 조정

이 흐름이 된다.

⸻

왜 이 방식이 중요한가

이 방식이 아니면 생기는 문제:
	•	AI 결과를 믿을 수 없다
