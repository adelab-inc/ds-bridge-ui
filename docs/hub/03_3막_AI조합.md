알겠어.

그럼 **3막을 처음부터 다시**,

앞의 맥락을 전제로 하지만 **이 문서 하나만 읽어도 이해되게**,

**실제 화면에서 무슨 일이 벌어지는지 중심으로**,

**톤앤매너 그대로** 정리해서 써줄게.

---

## **3막. AI가 ‘생성’이 아니라 ‘조합’을 한다 (Day 7–10)**

### **이 단계의 전제**

3막은 **AI가 갑자기 똑똑해지는 단계가 아니다.**

3막은 **사람이 이미 이해한 걸 AI가 대신 해주는 단계다.**

2막까지 사용자는 이미 손으로 확인했다.

- 이 디자인 시스템에 어떤 컴포넌트가 있는지
- 각 컴포넌트에 어떤 props가 있는지
- 어떤 조합이 자연스러운지
- 어떤 조합은 이 시스템에서 어색한지

그래서 질문이 이렇게 바뀐다.

> “이걸 매번 내가 직접 하지 말고,
> 

> AI가 대신 해줄 수는 없을까?”
> 

3막은 이 질문에 답하는 단계다.

---

### **먼저, AI가 하지 않는 것부터 명확히 하자**

3막에서 AI는 다음을 **하지 않는다.**

- ❌ 새로운 UI를 상상해서 그리지 않는다
- ❌ 디자인 시스템에 없는 컴포넌트를 만들지 않는다
- ❌ 정의되지 않은 props나 값을 만들어내지 않는다

즉,

> AI는 ‘창작자’가 아니다.
> 

> AI는 ‘조합자’다.
> 

AI의 역할은

**이미 존재하는 디자인 시스템 안에서**

**가장 그럴듯한 조합을 빠르게 만들어주는 것**이다.

---

### **그럼 AI는 정확히 무슨 일을 할까?**

AI가 하는 일은 딱 이거다.

> “이 디자인 시스템 안에 이미 있는 것들 중에서
> 

> 지금 목적에 맞는 컴포넌트 인스턴스들을 골라
> 

> 페이지 구조를 만들어준다.”
> 

여기서 중요한 점은:

- AI의 결과는 **새로운 화면**이 아니다
- AI의 결과는 **조합된 실행 상태**다

---

## **실제 화면에서 벌어지는 흐름**

### **1️⃣ 사용자가 한 줄을 입력한다**

화면 상단에는 입력창 하나가 있다.

```
What do you want to build?

> 로그인 페이지 만들어줘
```

이게 3막의 시작이다.

---

### **2️⃣ AI는 디자인 시스템을 “읽지 않고 참고한다”**

AI는 전체 디자인 시스템을 다 읽지 않는다.

필요한 부분만 본다.

- 이 DS에 어떤 컴포넌트들이 있는지
- 각 컴포넌트의 대표적인 story가 무엇인지
- 로그인이라는 목적에 맞는 컴포넌트가 무엇인지

즉,

> AI는 JSON을 ‘학습’하지 않고
> 

> ‘검색’해서 쓴다.
> 

---

### **3️⃣ AI의 결과는 ‘화면’이 아니라 ‘구성’이다**

AI가 내놓는 결과는 JSX도 아니고, 이미지도 아니다.

**이런 구조다.**

```
Login Page
 ├─ TextField (Email)
 ├─ TextField (Password)
 └─ Button (Primary)
```

각 항목은 명확하다.

- 어떤 컴포넌트인지
- 어떤 story를 기준으로 삼았는지
- 어떤 기본 상태(props)인지

이 결과는 **설명 가능한 데이터**다.

---

### **4️⃣ 이 조합이 바로 실행된다**

이 구성은 곧바로 화면에 올라간다.

```
┌───────────────────────────┐
│ [ Email Input ]           │
│ [ Password Input ]        │
│ [  Login Button  ]        │
└───────────────────────────┘
```

중요한 점은 이거다.

- 이 화면은 AI가 그린 그림이 아니다
- **스토리북에 이미 존재하던 컴포넌트들의 조합**이다

즉,

> AI는 그리지 않았고,
> 

> 골랐을 뿐이다.
> 

---

## **그래서 3막에서 사용자가 느끼는 감각**

이 시점에서 사용자는 이렇게 느낀다.

> “AI가 뭔가 새로 만든 게 아니네.”
> 

> “우리 시스템 안에서 맞는 걸 골라줬네.”
> 

> “그러면 이 결과는 믿을 수 있겠다.”
> 

이게 바로

**‘생성’과 ‘조합’의 결정적인 차이**다.

---

## **3막은 2막과 이렇게 이어진다**

AI가 만든 결과는 **끝이 아니다. 시작점**이다.

사용자는 다시 2막의 방식으로 돌아온다.

- 구성 트리에서 Button 선택
- variant 바꾸기
- size 바꾸기
- loading 상태 켜보기

흐름은 이렇게 된다.

```
AI → 기본 조합 제안
사람 → 실행 상태 미세 조정
```

AI는 방향을 잡아주고,

사람은 의미를 확정한다.

---

## **왜 이 방식이 중요한가**

이 방식이 아니면 생기는 문제는 분명하다.

- AI 결과를 신뢰할 수 없다
- “왜 이 버튼이 여기 있는지” 설명할 수 없다
- 회사에 가져가 쓰기 불안하다

하지만 이 방식이면:

- 모든 결과가 디자인 시스템 안에서 나온다
- 결과를 설명할 수 있다
- 다시 실행할 수 있다
- 링크로 공유해도 그대로 재현된다

즉,

> AI 결과가 ‘의견’이 아니라
> 

> ‘검토 가능한 초안’이 된다.
> 

---

## **3막의 성공 기준 (Definition of Done)**

이 단계는 아래가 되면 성공이다.

- 한 줄 프롬프트로 페이지 구성이 나온다
- AI가 고른 컴포넌트가 실제 DS에 존재한다
- 결과 화면이 즉시 실행된다
- 사용자가 2막 방식으로 다시 편집할 수 있다

---

## **3막의 역할은 여기까지다**

3막은 AI를 자랑하는 단계가 아니다.

3막은 **사람의 결정을 빠르게 시작하게 만드는 단계**다.

> “내가 할 수 있는 걸
> 

> AI가 먼저 해주네.”
> 

이 감각이 만들어지면,

이 플랫폼은 더 이상 실험 도구가 아니다.

---

> “이 흐름을
> 

> 우리 회사 디자인 시스템에도
> 

> 그대로 적용해도 되는 걸까?”
>
