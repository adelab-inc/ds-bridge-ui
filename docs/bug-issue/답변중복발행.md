## 증상

**재현 환경**: 거의 **프로덕션 환경에서만** 발생. 로컬 dev에서는 재현 빈도 낮음 (이유: 프로덕션 응답이 더 길고 느려 "이전 응답 스트리밍 중 새 질문" 조건이 자주 만들어지기 때문).

## 패턴 분석

서버가 자연스럽게 발행한 token delta를 `d1, d2, d3, d4, …`라 할 때, 프론트 상태에 실제로 append된 조각 순서는 다음과 같다:

```
d1 | (d1+d2) | d2 | d3 | (d3+d4) | d4 | d5 | (d5+d6) | d6 | d7 | (d7+d8) | d8
```

| 세그먼트 | 내용 | 정체 |
|---|---|---|
| 1 | "안녕…코" | d1 (delta) |
| 2 | "안녕…주" | **d1+d2 (combined)** |
| 3 | "드를…주" | d2 (delta) |
| 4 | "요 변경…정석 패턴" | d3 |
| 5 | "요 변경…Import 분" | **d3+d4 (combined)** |
| 6 | "에 따라…Import 분" | d4 |
| 7 | "리…수정" | d5 |
| 8 | "리…제거하였습니다" | **d5+d6 (combined)** |
| 9 | "하였…제거하였습니다" | d6 |
| 10 | ".\n4. …명시 및" | d7 |
| 11 | ".\n4. …준수했습니다." | **d7+d8 (combined)** |
| 12 | " 불필요한…준수했습니다." | d8 |

**규칙**: 연속된 두 개의 delta마다, 그 둘을 합친 "combined" 이벤트 1개가 추가로 끼어든다. 프론트는 이 세 종류를 구별 없이 `prev.text + payload.text`로 이어 붙여 중복이 누적된다.

## 프론트 코드 관점

### 문제의 지점

[apps/web/components/features/chat/hooks/use-chat-stream-lifecycle.ts:157-159](../../apps/web/components/features/chat/hooks/use-chat-stream-lifecycle.ts#L157-L159)

```typescript
if (payload.type === 'chat' && payload.text) {
  return { ...prev, text: prev.text + payload.text };
}
```

현재 `onChunk`는 들어오는 모든 `type:'chat'` payload를 **무조건 append** 한다.

- delta인지 snapshot인지 구분하는 필드가 payload에 없음 (`BroadcastChunkPayload` 참고, [apps/web/types/chat.ts:73-78](../../apps/web/types/chat.ts#L73-L78))
- overlap / 중복 수신 감지 없음
- 서버 발행 품질을 **전적으로 신뢰**하는 구조

## 원인 후보

> 로컬 dev(프로덕션 인프라 연결)에서 `[channel-event]`, `[onChunk]` 로그로 검증한 결과, **채널 중복 구독 / 에지 버퍼링 가설은 배제**되었다. 로그 관찰에서 새로 드러난 근본 원인이 있다.

### 🚨 확정 원인: `BroadcastChunkPayload`에 `message_id`가 없어 "이전 응답 누수" 발생

[apps/web/types/chat.ts:73-78](../../apps/web/types/chat.ts#L73-L78)

```typescript
export interface BroadcastChunkPayload {
  type: 'chat' | 'code';
  text?: string;
  content?: string;
  path?: string;
  // message_id 없음 ← 이것이 근본 문제
}
```

`BroadcastStartPayload`와 `BroadcastDonePayload`에는 `message_id`가 있지만, `BroadcastChunkPayload`에는 없다. 채널은 `room:{roomId}` 하나뿐이므로 **특정 chunk가 어떤 message의 것인지 식별할 수 없다**.

결과적으로 `onChunk`는 payload를 무조건 현재 `streamingMessage`에 append하게 되어, 다음 상황에서 **이전 응답의 잔여 chunk가 새 streamingMessage에 섞인다**:

- 이전 질문의 AI 응답이 아직 스트리밍 중
- 사용자가 새 질문을 보냄 → `handleSend`가 새 `streamingMessage` 생성
- 이전 응답의 나머지 chunk들이 도착 → 새 `streamingMessage`에 append → 텍스트 오염
- 뒤늦게 새 응답의 `onStart`가 도착
- 새 응답의 chunk들이 섞여서 append됨

**프로덕션에서 더 자주 발생하는 이유**: 프로덕션 AI 서버는 응답이 더 길고 느리며(Cold start, Cloud Run 재시도 등), "이전 응답이 아직 끝나지 않은 상태에서 새 질문" 상황이 잦다. 로컬 dev는 짧은 응답이나 빠른 완료로 이 조건이 덜 만들어진다.

### 로그 관찰로 배제된 가설

| 가설 | 상태 | 근거 |
|---|---|---|
| 채널 중복 구독 | ❌ 배제 | 모든 `[channel-event]`에서 `isStale: false` |
| Vercel/CDN 버퍼링 | ❌ 배제 | Supabase Realtime은 브라우저 ↔ Supabase 직접 WebSocket |
| React Strict Mode 이중 invocation | ❌ 배제 | 프로덕션 재현 |
| AI 서버 이중 publish (delta+snapshot) | ⚠️ 보류 | 로그에서 "2개 delta마다 combined 1개" 패턴이 재현되지 않음 — 원래 버그 스크린샷과 다른 경로일 가능성 있음. 서버 점검 시 동시에 확인 필요 |

### 로그 근거 요약

1. **테스트 1 (HMR 없이, 이전 응답 진행 중 상태)**:
   - `start` 이벤트가 chunk 10개 뒤에 늦게 도착
   - 첫 5개 chunk는 `prev is NULL, dropping` (streamingMessage가 null이라 드롭)
   - 다음 5개 chunk는 append됨 (handleSend 후 streamingMessage 생성된 이후)
   - 내용: 이전 질문의 응답(Badge type 패턴)이 새 질문의 streamingMessage에 침투
2. **테스트 2 (HMR 직후, 세션 깨끗)**:
   - `start` 먼저 → chunk 2개 → `code` chunk — 정상 흐름
   - 중복 없음
   - 차이점: HMR 리로드로 이전 응답 누수 조건이 초기화됨

## 즉시 확인 방법

> 프로덕션에서 재현되는 문제라 **프로덕션에서만** 수집 가능한 정보가 핵심. dev 환경에서 "안 나오니 괜찮다"고 멈추지 말 것.

### A. 프론트 로그 삽입

[apps/web/components/features/chat/hooks/use-chat-stream-lifecycle.ts:149](../../apps/web/components/features/chat/hooks/use-chat-stream-lifecycle.ts#L149)의 `onChunk` 진입부에 로그 추가:

```typescript
onChunk: (payload) => {
  console.log('[onChunk]', {
    ts: Date.now(),
    len: payload.text?.length,
    text: payload.text,
  });
  // 기존 로직...
}
```

수신 순서대로 payload를 나열하면 delta만 오는지, snapshot도 섞여 오는지 즉시 확인 가능.

### B. WebSocket 프레임 직접 확인

프로덕션 환경(로그인 상태)에서 DevTools → Network → WS → `realtime/v1/websocket?...` 연결 선택 → Messages 탭에서 `broadcast` 이벤트 payload의 `text` 필드 확인.

확인 포인트:
- **같은 내용이 두 프레임에 걸쳐 오는가** → 서버 이중 publish (원인 1 확정)
- **delta 1개가 정확히 2번 오는가** → 채널 중복 구독 (원인 2 확정)
- 프레임은 1번만 오는데 화면에만 중복 → 프론트 상태 업데이트 버그

### C. 프로덕션 AI 서버 로그 대조

AI 서버에서 Broadcast publish 시점에 로그를 남기고 있다면, 같은 `message_id`에 대해 publish가 몇 번 호출되는지 카운트:
- `message_id` 당 publish 호출 횟수 = chunk 수 → 정상
- `message_id` 당 publish 호출 횟수 > chunk 수 → 이중 publish 확정

호출 스택 / 호출자 정보를 함께 남기면 어떤 경로가 추가로 publish하는지 바로 특정 가능.

## 해결 방향

### ✅ 1차 — 프론트 start-gate 가드 (적용 완료)

서버 수정 없이 "이전 응답 누수"를 차단하는 방어 패치. 확정 원인에 대한 최단 경로 대응.

**구현**: [apps/web/components/features/chat/hooks/use-chat-stream-lifecycle.ts](../../apps/web/components/features/chat/hooks/use-chat-stream-lifecycle.ts)

| 추가한 것 | 동작 |
|---|---|
| `expectingStartRef` ref | `handleSend` 진입 시 `true`, `onStart` 수신 시 `false` |
| `onChunk`에서 gate 체크 | `expectingStartRef === true`이면 drop + `pre-start, dropping` 경고 |
| `onStart`에서 text 리셋 | `text/content/path`를 ""으로 초기화 (start 전에 들어온 chunk가 `prev.text`에 쌓여있을 수 있음) |
| `onDone`/`onError`/`handleTimeout`/POST 실패 | `expectingStartRef = false`로 원상복귀 |

**한계**: `onStart` 수신 후에 다른 응답의 chunk가 섞이는 **mid-stream 오염**은 프론트만으로는 해결 불가 (chunk에 `message_id`가 없어 구분 불가). → 2차 작업 필요.

### 🔜 2차 — 서버 작업: `BroadcastChunkPayload`에 `message_id` 추가 (근본 해결)

```typescript
interface BroadcastChunkPayload {
  type: 'chat' | 'code';
  message_id: string;   // ← 추가 필요
  text?: string;
  content?: string;
  path?: string;
}
```

프론트는 `payload.message_id !== activeMessageIdRef.current`면 무시하도록 추가 가드. 이 두 가지(서버 필드 추가 + 프론트 필터)가 적용되면 mid-stream 오염까지 완전 차단.

### 💡 3차 (선택) — `chunk_index` 추가로 프로토콜 강건화

재전송/중복 방어를 위해 `chunk_index`(증가 순번)까지 넣으면 네트워크 유실/재전송에도 dedup 가능. 필수는 아님.

```typescript
interface BroadcastChunkPayload {
  type: 'chat' | 'code';
  message_id: string;
  chunk_index: number;    // 응답 내 순번
  text?: string;
  ...
}
```

### 🔄 4차 (보조) — AI 서버의 delta vs snapshot 이중 발행 여부 점검

원래 버그 스크린샷("안녕하세요" 접두사 중복 + "2개 delta마다 combined 1개")은 "이전 응답 누수"만으로는 완전히 설명되지 않을 수 있음. 서버 publish 코드에서 token delta와 block accumulated가 동시에 같은 event로 발행되는 경로가 있는지 **추가 점검 권장**. 2차/3차 작업 중 자연스럽게 확인 가능.

### Phase 4 — 관측 인프라 구축 ✅ 완료

프로덕션에서 `pre-start dropping` 등 이상 이벤트를 수집하기 위한 경량 로깅 파이프라인. 기존 관측 도구(Sentry 등)가 없어 새로 구축.
