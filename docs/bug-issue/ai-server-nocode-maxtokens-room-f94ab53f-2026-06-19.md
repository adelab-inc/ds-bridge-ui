## 6. 권장 조치 (요약)

> ⚠️ **트레이드오프 주의**: 단순히 `gemini_max_output_tokens`를 다시 올리면 PR #150이 막 닫은 **thinking 폭주(무한 추론 → no-code)** 문제가 재발한다. "상한 원복"은 정답이 아니라 두 실패 모드 사이의 시소일 뿐 — 이 방처럼 대형 파일에선 상한을 올려도 600초 타임아웃으로 옮겨갈 뿐이다. 근본 해법은 **출력량 자체를 줄이는 것**.

1. **근본(우선)**: 전체 재출력 → **부분 수정 패치(search/replace 또는 unified diff)** 로 전환. 출력 토큰이 변경분에 비례 → 16k 상한 안에서도 안전, thinking 상한도 유지.
2. **구조**: 거대 단일 파일을 탭/그리드 단위 컴포넌트로 분할(파일당 출력 예산 확보).
3. **임시 완화(부분 패치 도입 전)**: 출력 상한을 동적으로 — `base_code` 크기에 비례해 `max_output_tokens`를 상향(예: `max(16384, est_output_tokens*1.3)`)하되, thinking 폭주는 thinking_level/별도 가드로 분리 통제. 단순 전역 상향(PR #150 원복)은 지양.
4. **UX/가드**: MAX_TOKENS로 인한 no-code를 "파일이 너무 커서 한 번에 재생성 불가"로 구체화하고, 마지막 정상 코드 보존/복원 경로 제공(현재는 일반 "실패" 문구 + 코드 유실).

## 8. 판정 — 서비스 설계 보완 사항인가, 유저 안티패턴인가

**판정: 압도적으로 서비스 설계 보완 사항.** 유저는 도구를 의도된 방식(설명 → 코드 → 수정 반복)대로 썼고 설계의 천장을 드러냈을 뿐이다.

**설계 측 결함 (주):**
1. **diff 없는 "전체 파일 재출력"** — 5글자 수정에도 115KB 전체를 재생성. 출력 비용이 *변경량이 아니라 전체 파일 크기에 비례*(O(n)). 정상 설계는 O(변경분). → 근본 결함.
2. **단일 파일 강제** — 7탭·다중 그리드가 하나의 `.tsx`에 누적되어 크기가 단조 증가. 위 1과 결합 시 반드시 출력 예산 초과.
3. **크기 인식 없는 고정 출력 상한** — 16,384 고정. "이 파일은 한 번에 재생성 불가"를 감지/경고하지 않고 일반 "실패"만 노출, 마지막 정상 코드 유실.
4. **두 실패 모드의 시소** — 상한 없으면 thinking 폭주, 상한 두면 출력 잘림. PR #150은 전자를 후자로 바꿨을 뿐 → "전체 재출력" 구조 자체가 제약이라는 신호.
5. **무의미한 재시도** — 결정적 출력 초과에는 재시도가 같은 벽에 막힘.

**유저 측 (부차, 그러나 비난 대상 아님):**
- 단일 방/파일에 105 프롬프트로 초복잡 엔터프라이즈 화면(해촉DB 배분, 50+컬럼 다중 그리드)을 구축 → 파일 비대화 가속.
- 다만 이는 실무 유저의 **자연스러운 사용**이며, 유저는 파일 크기 한계를 알 방법이 없다. "화면을 나눠 만드세요"는 유저가 실패로 추측할 게 아니라 **제품이 affordance로 제공**해야 할 몫.

**한 줄: "유저가 무리하게 썼다"가 아니라, 제품이 유저에게 권하는 복잡도를 설계가 감당하지 못한 것.** 우선순위는 §6 참조(최대 레버리지 = 전체 재출력 → diff 편집 전환).
